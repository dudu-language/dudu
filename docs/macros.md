# User-Defined Macros

[Dudu manual](https://dudulang.org/docs.html#macros) | Previous: [Native templates and macros](native-templates-and-macros.md) | Next: [Allocation and lifetimes](allocation-and-lifetimes.md)

Dudu macros are typed, additive declaration generators. A macro receives a
public Dudu declaration AST and returns ordinary Dudu declarations. It does not
receive source text, replace expressions, add grammar, or expose compiler
internals.

This guide covers the implemented macro system. The complete runnable fixture
is in [`tests/fixtures/macro_packages`](../tests/fixtures/macro_packages).

## Three Separate Mechanisms

Dudu uses decorator-shaped syntax for several things, but they are not all
user-defined macros.

| Mechanism | Example | Meaning |
| --- | --- | --- |
| Compiler-known decorator | `@operator("+")` | A language feature implemented by the compiler. |
| Imported native macro | `assert(value > 0)` | A C/C++ preprocessor macro scanned from a native header. |
| User-defined Dudu macro | `@derive(Debug)` | A Dudu function that generates typed declaration AST nodes at compile time. |

Compiler-known decorators include `@test`, `@operator`, `@constexpr`, and
`@extern_c`. They are fixed language syntax and cannot be redefined.

Imported C/C++ macros retain their native identity and are emitted through the
native preprocessor. See [Native templates and macros](native-templates-and-macros.md).

User-defined macros are the subject of this guide.

## Quickstart

Create a project containing these files:

```text
macro_demo/
  dudu.toml
  macros.dd
  main.dd
```

`dudu.toml`:

```toml
name = "macro_demo"
entry = "main.dd"
build_dir = "build"
```

`macros.dd` defines a macro using the public `dudu.ast` package:

```python
import dudu.ast as ast


def count_method(name: str, count: i32) -> ast.FunctionDecl:
    self_param = ast.Parameter(
        name="self",
        type=ast.reference_type(ast.named_type("Self")),
    )
    body = ast.return_statement(ast.int_expression(str(count)))
    return ast.FunctionDecl(
        name=name,
        parameters=[self_param],
        return_type=ast.named_type("i32"),
        body=[body],
    )


@macro
def Debug(item: ast.ClassDecl) -> ast.Expansion:
    count: i32 = 0
    for field in item.fields:
        count += 1

    out = ast.expansion()
    out.add_method(count_method("debug_field_count", count))
    return out
```

`main.dd` imports and applies it:

```python
from macros import Debug


@derive(Debug)
class Player:
    id: u64
    name: str
    hp: i32


def main() -> i32:
    player = Player(1, "Ada", 100)
    assert player.debug_field_count() == 3
    return 0
```

Inspect the generated declaration:

```bash
duc expand main.dd --show-origins
```

The relevant output is normal Dudu:

```python
# @Debug (macros.Debug) for main.Player
# invoked at main.dd:4:1
# macro defined at macros.dd:20:1
# source declaration at main.dd:5:1
# generated members
def debug_field_count(self: &Self) -> i32:
    return 3
```

Build, run, test, and emit C++ normally. Generated declarations pass through
the same resolver, semantic analyzer, lowering pipeline, and C++ emitter as
source declarations.

## Applying Macros

### Derives

`@derive(...)` applies one or more imported macros to a class or enum:

```python
from model_macros import Debug
from model_macros import Json


@derive(Debug, Json)
class Player:
    id: u64
    name: str
```

Every macro receives the original immutable declaration. `Json` does not see
members generated by `Debug`, and `Debug` does not see members generated by
`Json`. Expansion order cannot create a hidden transformation pipeline.

### Attached Macros

A macro may be attached directly when `derive` would be misleading:

```python
from model_macros import reflect


@reflect
class Transform:
    position: Vec3
    rotation: Quat
    scale: Vec3
```

Attached macros use the same typed input, additive output, caching, hygiene,
and diagnostics as derives.

### Typed Helper Attributes

A macro can export a named option schema:

```python
import dudu.ast as ast


class JsonOptions:
    name: str = ""
    skip: bool = False


@macro(attributes=JsonOptions)
def Json(item: ast.ClassDecl) -> ast.Expansion:
    count: i32 = 0
    for field in item.fields:
        options = ast.find_attribute(field.attributes, "Json")
        if options.has_value() and ast.bool_argument(options.value(), "skip", False):
            continue
        count += 1

    out = ast.expansion()
    out.add_method(count_method("json_field_count", count))
    return out
```

The same macro name becomes typed metadata on the source declaration:

```python
@derive(Json)
@Json(rename_all="camelCase")
class Player:
    id: u64

    @Json(name="displayName")
    name: str

    @Json(skip=True)
    cached_score: i32
```

The same-named decorator on the derived declaration configures that derive; it
does not invoke the macro a second time. This is independent of whether the
helper decorator appears before or after `@derive(...)`.

Unknown option names, wrong value types, duplicate options, and invalid
placements are compiler diagnostics before the worker runs. The fixture's
`Json` macro deliberately generates a small field-count method to test the
public API; it is not a bundled JSON serializer.

## Definition Signatures

`@macro` functions are ordinary Dudu functions with a checked public signature.
The first parameter selects the accepted declaration kind:

```python
@macro
def ClassMacro(item: ast.ClassDecl) -> ast.Expansion:
    return ast.expansion()


@macro
def EnumMacro(item: ast.EnumDecl) -> ast.Expansion:
    return ast.expansion()


@macro
def FunctionMacro(item: ast.FunctionDecl) -> ast.Expansion:
    return ast.expansion()
```

The return type must be `ast.Expansion`. Macro input and output use the public
SDK in `lib/dudu/ast.dd`, generated from the versioned protocol schema.

Macro modules may import normal Dudu helper modules. They compile as host tools,
separately from the target program.

The compiler computes macro-worker and target dependency reachability
separately. A helper used only by a macro stays out of target headers and object
files. A runtime codec or support module requested by generated code may be
needed by both graphs; it keeps one canonical type identity in both contexts.
Macro packages do not need wrapper headers or duplicated host/target type
declarations to share such a module.

Generated methods sometimes need a runtime module owned by the macro package.
Declare that dependency normally in the macro module, request it under a
private alias, and use the alias in generated AST:

```python
import dudu.ast as ast
import codec_runtime


@macro
def Codec(item: ast.ClassDecl) -> ast.Expansion:
    out = ast.expansion()
    out.require_module("codec_runtime", "_runtime")
    out.add_method(codec_method_using("_runtime"))
    return out
```

`require_module` does not load an arbitrary module or leak its names. The
compiler verifies that the macro definition module declares the dependency,
assigns the private alias a stable hygienic identity, and adds a qualified
target dependency. Users applying the macro do not import implementation
helpers themselves.

## Public AST

The read model includes:

- `ClassDecl`, `EnumDecl`, `FunctionDecl`, `FieldDecl`, and `ConstantDecl`
- fields, methods, enum variants, payload fields, and static data
- generic type and value parameters
- base classes and visibility
- structured `TypeRef`, `Expression`, and `Statement` values
- attributes, documentation, source ranges, and symbol identities

The builder model includes constructors for the same declarations plus common
helpers:

```python
ast.named_type("i32")
ast.reference_type(ast.named_type("Self"))
ast.int_expression("42")
ast.return_statement(ast.int_expression("42"))
ast.function_declaration(function)
ast.constant_declaration(constant)
ast.expansion()
```

An expansion can add:

```python
out.add_method(method)
out.add_field(field)
out.add_constant(constant)
out.add_sibling(ast.function_declaration(function))
out.add_implementation(implementation)
out.add_diagnostic(ast.warning(item.range, "message"))
out.require_module("codec_runtime", "_runtime")
```

The schema source and generated SDK are versioned. Macro packages never link
against parser, semantic-analysis, code-generation, or LSP classes.

## Complete Fixture Shapes

The package fixture validates nine public macros without compiler special
cases.

### Debug

`Debug` reads class fields and generates a method. This covers class input,
field traversal, parameters, a return type, statements, and member insertion.

```python
@derive(Debug)
class Player:
    id: u64
    name: str
```

### Json-Style Metadata

`Json` exports `JsonOptions`, reads typed field attributes, and skips marked
fields. A real serialization package can build encode/decode methods with the
same API.

```python
@derive(Json)
class Player:
    @Json(name="displayName")
    name: str

    @Json(skip=True)
    cached_score: i32
```

### String Enums

`StringEnum` receives `ast.EnumDecl`, traverses structured variants, and adds
an enum method:

```python
@derive(StringEnum)
enum Direction:
    North
    South
    East
    West
```

Payload variants are exposed with named fields, not token fragments.

### Reflection Metadata

An attached `@reflect` macro traverses a class and generates reflection-related
members. Reflection remains library code rather than a compiler-owned object
model.

### Command-Line Arguments

`Args` exports typed field options:

```python
@derive(Args)
class Options:
    input: str

    @Args(short="o")
    output: str

    @Args(default="False")
    verbose: bool
```

### Binary Schemas

`BinarySchema` reads integer helper arguments and generates schema metadata:

```python
@derive(BinarySchema)
class Packet:
    @BinarySchema(bits=16)
    kind: u16

    @BinarySchema(bits=32)
    sequence: u32
```

### Native Export Tables

`plugin_exports` attaches directly to a class, reads class-level options, and
adds a sibling function:

```python
@plugin_exports(prefix="demo")
class PluginApi:
    def initialize(self) -> i32:
        return 0

    def tick(self, dt: f32):
        pass
```

The fixture generates `demo_export_count()` as a small proof of sibling
generation. A binding package can use the same shape to generate C ABI entry
points or registration tables.

### Diagnostics

`Warn` returns a public macro diagnostic:

```python
@macro
def Warn(item: ast.ClassDecl) -> ast.Expansion:
    out = ast.expansion()
    out.add_diagnostic(ast.warning(item.range, "showcase warning from public macro API"))
    return out
```

Warnings, errors, notes, related ranges, and source origins cross the worker
protocol as structured values.

## Additive Rules

A macro may add members, sibling declarations, abstract-contract
implementations, metadata declarations, and diagnostics.

A macro cannot:

- remove, rename, or replace source declarations
- rewrite source function bodies
- change a source field type
- introduce another macro invocation
- add grammar or expression syntax
- return source strings or token streams
- install type-inference or overload-resolution rules

Generated names are checked against source and other generated names. A
conflict reports the macro invocation and the conflicting declaration.

Private generated helpers use stable hygienic identities. They cannot capture
source locals accidentally. Public names remain explicit because source code
must be able to call them.

## Diagnostics And Recovery

Macro failures are compiler diagnostics, not worker crashes printed to a
terminal. The diagnostic chain can include:

- the macro invocation
- the source declaration or field that caused the error
- the macro definition
- notes emitted by the macro

Malformed responses, oversized protocol messages, timeouts, and worker exits
are bounded. A failed worker is discarded; a later request starts a clean
worker rather than reusing poisoned process state.

## Expansion Inspection

The lower-level compiler provides three inspection forms:

```bash
duc expand src/main.dd
duc expand src/main.dd --macro Json
duc expand src/main.dd --show-origins
```

Write output to a file with `-o`:

```bash
duc expand src/main.dd --show-origins -o build/expanded.dd
```

This displays generated Dudu, not generated C++. Use the normal C++ emission
commands when inspecting lowering and ABI output.

## Packages And Imports

Macro definitions use normal Dudu modules and package resolution:

```python
from project_macros import Debug
from project_macros import Json
```

Path and Git dependencies use the same `dudu.toml` and lockfile model as other
Dudu packages. Macro package source, helper modules, native host dependencies,
compiler version, host toolchain settings, and public SDK version participate
in the worker binary cache key.

Macro dependencies are trusted build dependencies. Review and pin them as
carefully as compiler plugins or native build scripts.

## Capabilities

Deterministic macros need no host capabilities. A macro that reads an external
schema or environment value declares the narrow access in `dudu.toml`:

```toml
[macro.capabilities]
fs.read = ["schemas/**"]
env.read = ["TARGET"]
```

The host API is imported separately:

```python
import dudu.macro as host


schema = host.read_text("schemas/messages.json")
target = host.read_env("TARGET")
```

Accepted capabilities are:

- `fs.read`
- `fs.write`
- `env.read`
- `process`
- `network`
- `clock`
- `random`

The SDK exposes `read_text`, `write_text`, `read_env`, `run`,
`require_network`, `clock_ns`, and `random_u64`.

File reads and environment values are captured in expansion cache validation.
Writes, process execution, network access, time, and randomness make a macro
non-cacheable. Such macros must be named explicitly:

```toml
[macro]
allow_non_cacheable = ["GenerateAssets"]

[macro.capabilities]
process = []
clock = true
```

Undeclared access is rejected.

## Caching And Workers

Each macro package compiles to a host-native worker. One persistent worker
serves compatible invocations within a compiler or LSP session. Startup is not
paid once per declaration.

There are two content-addressed caches:

1. The worker binary cache tracks macro source, helper dependencies, SDK and
   protocol versions, host compiler settings, and capabilities.
2. The expansion cache tracks worker identity, macro entry point, canonical
   declaration AST, helper attributes, compile-time values, and captured
   external inputs.

An unrelated source edit executes zero unchanged macros. Changing one helper
attribute invalidates that declaration's expansion rather than every use of the
macro package.

Use `--timings` to inspect the work:

```bash
duc expand src/main.dd --timings -o build/expanded.dd
```

Reported categories include planning, declaration bridging, package build,
worker startup, protocol transport, execution, cache reads and writes,
validation, hygiene, collection, and merge.

The explicit benchmark suite is outside the fast correctness loop:

```bash
scripts/bench_compiler.sh --suite macros --samples 5 --build-type Release
```

The release matrix includes 1, 100, 1,000, and 10,000 invocations, generated
payloads through one million public AST nodes, incremental edits, a handwritten
equivalent, worker RSS, and an informational Rust procedural-macro comparison.

On the reference machine after the indexed merge work, 10,000 cached Debug
derives measured about 1.07 times the equivalent handwritten frontend fixture.
Machine-specific output remains under `build/bench_compiler` and is not a
portable timing promise.

## Editor Behavior

The compiler and LSP share the same macro registry, workers, expansion cache,
and generated AST. In supported editors:

- macro imports and invocations have semantic highlighting
- hover shows the accepted declaration kind, option schema, signature, and docs
- go-to-definition opens the macro definition
- references and rename use macro symbol identity rather than spelling
- helper option completion and signature help come from the typed schema
- generated members participate in completion, hover, navigation, and errors
- generated symbols identify their macro invocation and definition origin
- macro diagnostics remain structured editor diagnostics

The LSP does not run a separate text-expansion implementation.

## What To Use Instead Of Unsupported Macros

| Unsupported form | Use instead |
| --- | --- |
| Expression macro | Function, generic function, operator overload, or `@constexpr`. |
| Control-flow macro | Function, class, iterator, or ordinary control flow. |
| Source substitution | Constant, function, generic, or imported native macro. |
| Custom grammar | Normal Dudu declarations and calls. |
| Function-body rewrite | Explicit helper function or generated sibling/member declaration. |
| Type-system plugin | Generic constraints and normal Dudu types. |
| C preprocessor behavior | Import the C/C++ header at the native boundary. |

The absence of quote/splice syntax is deliberate. Dudu source is not a Lisp
data representation, and source-string generation would lose type structure,
origins, hygiene, and editor behavior.
